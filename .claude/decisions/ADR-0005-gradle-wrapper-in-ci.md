# ADR-0005: Gradle Wrapper Strategy — Conditional Bootstrap in CI

**Date**: 2026-05-22
**Updated**: 2026-05-22 (session 9)
**Status**: Accepted (superseded partially — gradlew/gradlew.bat now committed; jar pending)
**Deciders**: Session 8 (initial), Session 9 (updated)

---

## Context

`android/gradlew`, `android/gradlew.bat`, and `android/gradle/wrapper/gradle-wrapper.jar`
were not committed to the repository. The CI workflow introduced in TASK-017
references `./gradlew assembleRelease` but the file did not exist.

Standard Android practice is to commit all three wrapper files. The text files
(`gradlew`, `gradlew.bat`) can be authored manually or generated. The binary
`gradle-wrapper.jar` (~60KB) must be generated by a Gradle installation.

---

## Decision (updated — session 9)

**Phase 1 (session 8)**: Added unconditional CI step to generate the full wrapper.
**Phase 2 (session 9)**: Split into:
1. `gradlew` + `gradlew.bat` — committed directly (text files, stable across versions)
2. `gradle-wrapper.jar` — still missing; CI step made conditional:

```yaml
- name: Ensure Gradle Wrapper jar (bootstrap if missing)
  working-directory: android
  run: |
    if [ ! -f gradle/wrapper/gradle-wrapper.jar ]; then
      echo "gradle-wrapper.jar not in repo — generating via system gradle (TD-CI-001)"
      gradle wrapper --gradle-version 8.3 --distribution-type all
    else
      echo "gradle-wrapper.jar present — skipping generation"
    fi
    chmod +x gradlew
```

**Phase 3 (pending TD-CI-001)**: Run `bash scripts/commit_gradle_wrapper.sh` locally →
commit jar → CI step becomes a trivial no-op → step can be removed entirely.

---

## Alternatives considered

**A. Commit all three files at once** (ideal): Blocked because `javac`/`gradle` are not
available in the current execution environment. Deferred to local run.

**B. Keep unconditional generate step forever**: Adds ~15–30s on cache miss every run.
Non-standard. Rejected — conditional step is strictly better.

**C. Use `gradle/actions/setup-gradle@v3`**: Official GitHub Action. Adds wrapper
validation. Requires `gradlew` to already exist for most modes. More overhead.

---

## Consequences

**Current state** (jar not yet committed):
- CI adds ~15–30s for wrapper generation on cache miss (jar absent → generates)
- On cache hit, Gradle download is skipped but generate step still runs (trivial)
- `gradlew` + `gradlew.bat` are committed; developers can run locally if they
  generate the jar first (same `gradle wrapper` command)

**After TD-CI-001 resolved** (jar committed):
- CI bootstrap step runs in <1s (jar present → just does `chmod +x gradlew`)
- `./gradlew` works immediately after checkout with no external tooling
- Fully reproducible, fully standard Android project structure

---

## Risks

| Risk | Level | Mitigation |
|------|-------|------------|
| `gradle` not on ubuntu-latest runner | LOW | ubuntu-latest includes Gradle; conditional step only runs if jar absent |
| Version mismatch (system gradle vs 8.3) | LOW | We pass `--gradle-version 8.3` explicitly |
| jar never committed | LOW | `scripts/commit_gradle_wrapper.sh` makes it one command |
